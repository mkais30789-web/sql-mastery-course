Order of Operations

SQL doesn't execute top to bottom. It follows a specific order. Understanding this helps write better queries.

1
FROM/JOIN
2
WHERE
3
GROUP BY
4
HAVING
5
SELECT
6
ORDER BY
7
LIMIT

Example

SQL
-- Execution order:
SELECT category, SUM(sales)
FROM orders                  -- 1. Load
WHERE status='active'     -- 2. Filter rows
GROUP BY category          -- 3. Group
HAVING SUM(sales)>1000  -- 4. Filter groups
ORDER BY SUM(sales)      -- 6. Sort
LIMIT 10;                  -- 7. Top 10

WHERE filters BEFORE grouping, HAVING filters AFTER. That's why you can't use aliases in WHERE!

📝 Summary

  • FROM/JOIN defines source data
  • WHERE filters individual rows (before GROUP BY)
  • HAVING filters grouped results (after GROUP BY)

WHERE vs HAVING

WHERE

Filters BEFORE grouping. Works on rows. Cannot use with aggregates.

HAVING

Filters AFTER grouping. Works on groups. Can use with aggregates.

Example

SQL
SELECT customer_id, COUNT(*) AS orders, SUM(amount) AS total
FROM orders
WHERE amount>50        -- Filter: orders over $50
GROUP BY customer_id
HAVING SUM(amount)>1000; -- Filter: over $1000 total

Use WHERE before grouping when possible - it's faster!

📝 Summary

  • WHERE = row filter (before GROUP BY)
  • HAVING = group filter (after GROUP BY)
  • WHERE is faster - filters less data to group

JOINs

TypeReturns
INNER JOINOnly matching rows
LEFT JOINAll left + matches
RIGHT JOINAll right + matches
FULL OUTERAll rows from both
CROSS JOINEvery combination

INNER JOIN

SQL
SELECT o.order_id, c.name, o.amount
FROM orders o
INNER JOIN customers c ON o.customer_id=c.id;

LEFT JOIN

SQL
SELECT c.name, COUNT(o.id) AS orders
FROM customers c
LEFT JOIN orders o ON c.id=o.customer_id
GROUP BY c.id;

Multiple JOINs

SQL
SELECT o.id, c.name, p.name, r.name
FROM orders o
JOIN customers c ON o.customer_id=c.id
JOIN products p ON o.product_id=p.id
JOIN regions r ON c.region_id=r.id;

Always include ON condition! Without it you get Cartesian product.

📝 Summary

  • INNER - Only matches
  • LEFT - All left + matches
  • RIGHT - All right + matches
  • CROSS - Every combination

UNION

UNION

Removes duplicates. Slower.

UNION ALL

Keeps all rows. Faster.

Example

SQL
SELECT id, name, 'Active' AS status FROM users WHERE active=TRUE
UNION
SELECT id, name, 'Inactive' AS status FROM users WHERE active=FALSE;

Both queries must have same column count and types.

📝 Summary

  • UNION removes duplicates, UNION ALL keeps them
  • UNION ALL is faster - use when possible

Subqueries

A query inside another query. Like a temporary calculation.

In WHERE

SQL
SELECT name, price
FROM products
WHERE price > (SELECT AVG(price) FROM products);

In FROM

SQL
SELECT region, avg_sales
FROM (
    SELECT region, AVG(sales) AS avg_sales
    FROM stores GROUP BY region
) stats
WHERE avg_sales > 10000;

In SELECT

SQL
SELECT name, amount, (
    SELECT AVG(amount) FROM orders o2
    WHERE o2.customer_id=o1.customer_id
) AS customer_avg
FROM orders o1;

📝 Summary

  • WHERE: Filter based on calculated value
  • FROM: Use as temporary table
  • SELECT: Add calculated column

CTEs

Named subquery. Define once, use multiple times. Much cleaner than nested queries!

Basic Syntax

SQL
WITH cte_name AS (
    SELECT * FROM table WHERE condition
)
SELECT * FROM cte_name;

Multiple CTEs

SQL
WITH orders_sum AS (
    SELECT customer_id, SUM(amount) AS total
    FROM orders GROUP BY customer_id
),
ranked AS (
    SELECT *, RANK() OVER(ORDER BY total DESC) AS r
    FROM orders_sum
)
SELECT * FROM ranked WHERE r <= 10;

CTEs also enable recursive queries for hierarchical data!

📝 Summary

  • Named, reusable query blocks
  • Chain multiple CTEs together
  • More readable than subqueries
